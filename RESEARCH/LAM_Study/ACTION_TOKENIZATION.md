# Action Tokenization: Theoretical Framework\n\n## 1. Problem Statement\nStandard LLMs generate text. LAMs generate **actions**. The mapping from action space to token space is the critical interface.\n\n### 1.1 Flat Tokenization (The Baseline Problem)\n- **Method**: Direct text generation (e.g., `click(x=10, y=20)`).\n- **Flaw**: The model treats the action as a string generation task, ignoring the discrete nature of API calls. This leads to syntax errors (e.g., `click(x=10, y=20`) and semantic drift.\n- **Metric Failure**: High token-level loss despite correct semantic intent.\n\n### 1.2 Hierarchical Semantic Tokenization (Our Proposal)\n- **Concept**: Decompose the action into a sequence of hierarchical tokens.\n- **Structure**:\n  - **Root**: Environment/Task Domain (e.g., `WEB`, `OS`, `MOBILE`).\n  - **Branch**: API Module or UI Component (e.g., `INPUT`, `NAVIGATION`, `SCROLL`).\n  - **Leaf**: Specific Action Primitive (e.g., `CLICK`, `TYPE`, `ENTER`).\n  - **Payload**: Arguments (key-value pairs).\n\n### 1.3 Key Innovation: Reward-Aligned Embeddings\n- We propose mapping action leaves to a continuous vector space.\n- **Hypothesis**: Actions with similar reward profiles should be close in this space.\n- **Benefit**: Allows for *continuous RL* over a discrete action set (hybrid approach).\n\n## 2. Comparison with SOTA\n- **ToolBench**: Uses API schema descriptions. (Weakness: relies heavily on prompt engineering).\n- **AgentBench**: Evaluates code generation. (Weakness: assumes perfect syntax execution).\n- **Proposed**: Explicit tokenization layer + RL alignment.\n\n## 3. Implementation Strategy\n1. Define the ontology of actions for the first benchmark (Web).\n2. Train a lightweight tokenizer (or fine-tune existing BPE) on action traces.\n3. Implement the hierarchical wrapper in Gymnasium.
